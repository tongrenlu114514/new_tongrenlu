## Context

当前音乐库页面 (`index.html` -> `#musicPage`) 使用原生 JavaScript + jQuery 实现，包含搜索、筛选、分页等基础功能。主要痛点：

1. **搜索体验不足**: 无实时建议、无历史记录、无防抖优化
2. **筛选标签硬编码**: 当前 6 个标签（全部/最新/热门/摇滚/电子/古典）为静态 HTML，与数据库标签表 (`m_tag`) 无关联
3. **排序功能缺失**: 后端 API 支持排序但前端未暴露
4. **分页样式简陋**: 仅显示数字按钮，无跳转输入、无总数显示
5. **加载反馈弱**: 仅显示文字"搜索中..."，缺少骨架屏

现有技术栈：Spring Boot 3.4.3 + MyBatis Plus + 原生前端 (jQuery 3.7.1)

## Goals / Non-Goals

**Goals:**
- 提升搜索效率：实时建议 + 历史记录
- 实现动态筛选：标签从后端加载
- 完善排序功能：多维度排序选项
- 改善视觉反馈：骨架屏 + 优化分页

**Non-Goals:**
- 不重构为 Vue/React 等前端框架
- 不改变现有 API 响应结构
- 不新增用户登录相关功能
- 不实现高级筛选面板（多条件组合筛选）

## Decisions

### 1. 搜索建议实现方案

**选择**: 前端防抖 + 后端模糊搜索

**理由**:
- 防抖 300ms 避免频繁请求，保护后端
- 复用现有 `api/music/search` 接口，新增 `limit=5` 参数返回建议列表
- 使用 `localStorage` 存储最近 10 条搜索历史

**替代方案**:
- 前端预加载全量数据 → 内存占用过高，不适用大数据量
- Elasticsearch 全文搜索 → 引入复杂度过高

### 2. 动态筛选标签方案

**选择**: 页面加载时调用 `api/music/tags` 获取热门标签

**理由**:
- 现有 `m_tag` 表已维护标签数据，直接复用
- 返回访问量 Top 10 标签，保持筛选精准性
- 保持单标签选择模式，交互简单

**替代方案**:
- 多标签组合筛选 → 后端逻辑复杂，暂不实现
- 全量标签展示 → 标签过多影响用户体验

### 3. 排序功能方案

**选择**: URL 参数传递排序字段

**理由**:
- 后端 `api/music/search` 已支持 `orderBy` 参数
- 前端添加排序下拉框，传递 `orderBy=accessCount|publishDate|title`
- URL 状态可分享，支持刷新保持状态

### 4. 骨架屏实现方案

**选择**: CSS 纯实现 + 渐变动画

**理由**:
- 无额外依赖
- 使用 `@keyframes shimmer` 实现加载动画
- 骨架卡片结构与真实卡片一致，避免布局抖动

## Risks / Trade-offs

| 风险 | 缓解措施 |
|------|----------|
| 搜索建议接口响应慢 | 设置 500ms 超时，失败时静默降级 |
| localStorage 被禁用 | try-catch 包裹，失败时跳过历史功能 |
| 标签数据为空 | 保留默认"全部"标签作为兜底 |
| 分页跳转输入校验 | 限制输入范围 1-totalPages |
