## Context

同人录当前主页采用传统的单页面结构，包含：
- Hero 区域：图片画廊风格展示专辑封面
- 导航栏：显示专辑数量统计
- 底部播放器
- 页面切换到音乐库

现有技术栈：
- 前端：原生 HTML/CSS/JavaScript + jQuery 3.7.1 + Bootstrap CSS
- 后端：Spring Boot 3.4.3 + MyBatis Plus
- 现有组件：`components/home/` 目录下的模块化 JS 文件

设计约束：
- 保持与现有播放器组件的兼容性
- 使用现有的 CSS 变量系统 (`--primary-color`, `--secondary-color` 等)
- 不引入新的前端框架

## Goals / Non-Goals

**Goals:**
- 提升首页视觉吸引力，使用图片画廊风格展示专辑封面
- 为用户提供直观的内容发现入口
- 优化移动端体验
- 保持代码可维护性和模块化

**Non-Goals:**
- 不重新设计音乐库页面
- 不修改播放器核心功能
- 不引入新的前端框架（如 React/Vue）
- 不修改后端数据库结构

## Decisions

### 1. 页面布局结构

**决策：采用图片画廊风格的 Hero 区域 + 最新专辑轮播**

```
┌─────────────────────────────────────┐
│  导航栏 (固定) + 专辑数量统计        │
├─────────────────────────────────────┤
│  ┌──────┬────┬────┬────┬────┐       │
│  │      │    │    │    │    │       │
│  │ 大图 ├────┼────┼────┤ 竖 │       │
│  │      │    │    │    │ 图 │       │
│  ├──────┴────┴────┴────┤    │       │
│  │   宽图    │    │大图│    │       │
│  ├────┬────┬────┬────┴────┴────┐   │
│  │    │    │    │    │    │    │   │
│  │    │    │    │    │    │    │   │
│  └────┴────┴────┴────┴────┴────┘   │
│        图片画廊 Hero 区域            │
├─────────────────────────────────────┤
│        最新专辑轮播                  │
│   (横向滚动卡片式展示)               │
├─────────────────────────────────────┤
│          页脚                        │
└─────────────────────────────────────┘
```

**理由**：图片画廊风格视觉冲击力强，能充分展示专辑封面的艺术性，同时保持页面简洁。移除中心覆盖层让用户可以直接点击封面交互。

**替代方案**：传统 Hero 区域 - 被否决，因为视觉单调，无法展示内容库的丰富性。

### 2. Hero 画廊实现

**决策：使用 CSS Grid 实现画廊布局，15 个位置混合大图/宽图/竖图**

布局特点：
- 15 个专辑封面位置
- 大图（span 2x1）、宽图（span 2x1）、竖图（span 1x2）混合排列
- 通过 API 获取随机专辑，每 30 秒自动刷新
- 悬停时显示专辑标题
- 无中心覆盖层，封面可直接点击

**理由**：CSS Grid 原生支持复杂布局，性能好，无需 JavaScript 计算位置。

### 3. 导航栏统计显示

**决策：在导航栏右侧显示专辑数量统计**

显示内容：
- 专辑数量徽章（渐变背景圆角样式）
- 格式：「X 张专辑」

**理由**：将统计信息移至导航栏，释放 Hero 区域空间，让封面更突出。

### 4. 轮播组件实现

**决策：使用纯 CSS + jQuery 实现轮播，不引入第三方库**

实现方式：
- CSS `scroll-snap` 实现平滑滚动
- jQuery 处理自动播放和导航逻辑
- 使用现有 FontAwesome 图标

**理由**：保持轻量，避免额外依赖，与现有代码风格一致。

### 5. 响应式断点

**决策：使用多个断点适配不同设备**

```css
/* 大屏幕 */
@media (min-width: 1601px) { }  /* 音乐库 8 列 */

/* 中等屏幕 */
@media (max-width: 1600px) { }  /* 音乐库 6 列 */

/* 桌面端 */
@media (min-width: 1025px) { }

/* 平板端 */
@media (max-width: 1024px) { }

/* 移动端 */
@media (max-width: 768px) { }
```

**理由**：与现有样式保持一致，简化维护，音乐库限制每行最多 8 个专辑。

### 6. API 数据获取

**决策：新增随机专辑 API，复用统计 API**

API 使用：
- `GET /api/music/random-albums?count=15` - 随机专辑列表（用于画廊）
- `GET /api/music/album-stats` - 专辑统计（导航栏显示）
- `GET /api/music/search` - 专辑列表（用于最新专辑轮播）

**理由**：新增 random-albums 接口确保每次获取的都是随机数据，避免客户端打乱。

## Risks / Trade-offs

**[风险 1] 页面加载性能**
- 问题：画廊需要加载多张图片
- 缓解：图片懒加载、使用 CDN 加速、骨架屏

**[风险 2] 移动端兼容性**
- 问题：画廊在小屏幕上效果需要简化
- 缓解：移动端使用更简单的网格布局（2x3 或 3x3）

**[风险 3] 现有代码冲突**
- 问题：新样式可能影响现有组件
- 缓解：使用 CSS 类名前缀 `.home-`，避免全局污染

**[权衡] 视觉丰富 vs 加载速度**
- 选择：画廊风格视觉优先，但限制图片数量
- 原因：首屏视觉体验是关键

## Migration Plan

### 阶段 1：基础设施
1. 创建新的 CSS 文件 `assets/css/home-redesign.css`
2. 创建新的 JS 组件文件
3. 不影响现有页面

### 阶段 2：逐步替换
1. 在 `index.html` 中引入新样式和脚本
2. 替换 Hero 区域为画廊风格
3. 添加最新专辑轮播

### 阶段 3：清理
1. 移除不再使用的旧样式
2. 更新组件引用
3. 测试所有功能

### 回滚策略
- 保留原有 `style.css` 中的样式作为备份
- Git 版本控制确保可回滚
